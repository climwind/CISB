**Title**: Fix infinite loop in llist macros due to Clang optimization assuming member address non-NULL.

**Issue**: Clang optimizes `&pos->member != NULL` to always true when member offset > 0, breaking loop termination and causing infinite iteration.

**Tag**: Compiler workaround / bug fix.

**Purpose**: Replace direct member address check with integer arithmetic to prevent Clang's incorrect optimization.

---

### Step-by-Step Analysis:

1. **Key Variables/Functionality**: The macros `llist_for_each_entry` and `llist_for_each_entry_safe` use `&pos->member != NULL` as loop condition. `pos` is derived via `container_of()` which can return non-NULL pointers with NULL member addresses.

2. **Compiler Behavior**: Clang assumes that taking address of a struct member with positive offset cannot yield NULL (since object base must be non-NULL due to pointer arithmetic UB), optimizing the check away.

3. **Pre/Post Compilation**: Without optimization, condition evaluates at runtime; with optimization, condition becomes constant true, changing loop from terminating to infinite.

4. **Security Implications**: Infinite loops cause denial of service (hang), a direct security impact in kernel context.
