**Title**: Prevent interim PTE states by enforcing atomic writes with WRITE_ONCE().

**Issue**: Compiler optimization could split PTE assignments into multiple instructions, creating transient non-present entries that speculative execution (e.g., L1TF) might exploit.

**Tag**: security fix

**Purpose**: Replace direct assignments to page-table entries with WRITE_ONCE() to ensure atomic writes and eliminate interim states.

---

### Step-by-Step Analysis:

1. **Key Variables/Functionality**: Direct assignments to page-table entry pointers (`*ptep`, `*pmdp`, `*pudp`, etc.) in x86 MMU code. These are used to set or clear PTEs, PMDs, PUDs, etc.

2. **Compiler Behavior**: The compiler may optimize the assignment of a large structure (like a PTE) by writing it in multiple instructions (e.g., separate writes for different parts of the structure). This is a permissible optimization since the C standard does not guarantee atomicity for such assignments.

3. **Pre/Post Compilation**: Before compilation, the code assumes the assignment occurs atomically. After compilation, without WRITE_ONCE(), the assignment might be split, creating intermediate states where the PTE is partially written (e.g., non-present but with other fields set). With WRITE_ONCE(), the compiler is forced to treat the assignment as a single, atomic write.

4. **Security Implications**: Interim non-present PTEs could be speculatively accessed by the CPU, potentially leaking data via side-channel attacks like L1TF. This is an indirect security implication related to speculative execution.
